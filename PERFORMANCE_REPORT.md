| **隧道额外开销** | **+84 ms（约 8× 原始网络延迟）** |

**分析：** 隧道额外开销主要来自：
1. TLS 握手（WebSocket over WSS）
2. 服务端 asyncio 协程调度
3. 客户端本地转发到目标端口

对于 SSH、HTTP 管理等非实时场景完全可接受；对延迟敏感的游戏或 VoIP 类应用则偏高。

## 六、已知问题

### 6.1 扫描器触发并发错误（严重性：低）

**现象：** 服务器日志出现 `RuntimeError: cannot reuse already awaited coroutine`，来自 `client_connected_cb`，由外部 IP（58.61.248.5）大规模扫描 TCP 端口触发。

**影响：** 错误被 asyncio 捕获，**不影响正常连接**（同时段合法 TCP 隧道连接全部成功）。

**根因：** asyncio `start_server` 的回调在极端并发下存在协程复用问题，是框架级边界情况。

**缓解建议：**
```
# iptables 限速：单 IP 每秒不超过 5 个新连接
iptables -A INPUT -p tcp --dport 2201:2300 -m state --state NEW \
  -m hashlimit --hashlimit-name tcp_tunnel --hashlimit-above 5/sec \
  --hashlimit-mode srcip --hashlimit-burst 10 -j DROP
```

### 6.2 ICMP 被屏蔽

服务器防火墙屏蔽 ICMP，导致无法使用 `ping` 测量网络延迟，已改用 TCP 方式替代。不影响功能。

## 七、总体评估

| 维度 | 评级 | 说明 |
|------|------|------|
| 连接稳定性 | ★★★★★ | 连续在线 589s+，心跳机制工作正常 |
| TCP 隧道可用性 | ★★★★★ | 50 并发 100% 成功，实际场景绰绰有余 |
| 延迟表现 | ★★★☆☆ | 隧道 RTT ~96ms，比直连高 8×，满足 SSH/HTTP 使用 |
| 吞吐量 | ★★★★☆ | 44 MB/s，受限于服务器出口带宽 |
| 资源效率 | ★★★★★ | 45 MB 内存，0.1% CPU，极其轻量 |
| 并发处理 | ★★★★☆ | 50 并发全成功，有扫描时偶见错误日志但不影响使用 |

**总评：** 对于个人/家庭/NAS 内网穿透场景，easy_vpn 性能完全满足需求。主要瓶颈是隧道引入的额外延迟（~84ms），适合对延迟不敏感的管理类流量（SSH、Web 管理、NAS 访问等）。如需降低延迟，可考虑使用更近的服务器节点或优化服务端 asyncio 任务调度。
