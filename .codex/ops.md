# Codex Ops Guide

本文件定义 Codex 在本仓库中的操作准则。
项目背景、架构速记和长期记忆请查看 `.codex/memory.md`。

## 1. 服务器边界

- 测试环境服务器：`119.91.235.65:22022`
- 生产环境服务器：`43.163.90.149:22022`
- 未经用户明确授权，禁止操作生产环境。
- 默认只允许连接、检查和修改测试环境。
- 如果用户提到“服务器”但未明确说明，默认指测试环境，不要自行切到生产环境。

## 2. 发布与更新原则

- 服务器上的服务更新，必须通过 GitHub 拉取代码。
- 禁止使用 `rsync` 进行代码同步发布。
- 禁止把本地未提交文件直接覆盖到服务器。
- 优先使用项目内现成的一键部署脚本，不要临时拼装发布流程。
- 如果需要部署的内容尚未进入 GitHub，应先停止部署并向用户说明原因。

## 3. 服务器工作树处理规则

- 如果服务器上的仓库工作树是 dirty，不能直接硬做 `git pull`。
- 不要擅自执行以下破坏性操作：
  - `git reset --hard`
  - `git checkout -- <file>`
  - 删除未提交文件
  - 覆盖服务器上的手工改动
- 遇到 dirty 工作树时，先判断这些改动是否属于有效线上更改。
- 如果无法确认改动来源或是否允许丢弃，先暂停并向用户确认。
- 更稳的替代方案包括：
  - 新建干净目录，从 GitHub 重新 clone
  - 复用现有 `.env` 和必要运行配置
  - 在新目录执行部署脚本

## 4. 部署前检查

执行部署前，至少确认以下项目：

- 目标服务器是否正确
- 当前操作是否得到授权
- 目标目录是否为正确项目目录
- Git remote 是否指向正确仓库
- `.env` 是否存在且已配置
- Docker 是否可用
- Docker Compose 是否可用
- Nginx 是否可用
- 项目内一键部署脚本是否存在且与当前环境匹配

## 5. 部署执行规则

- 优先使用项目内一键部署脚本。
- 如果测试环境不需要正式 SSL，可优先使用测试参数，例如 `--skip-ssl` 或项目文档中的测试模式参数。
- 如果脚本内部自带 `git pull`，也要先确认其执行目录和当前工作树状态是否安全。
- 若脚本会修改 Nginx、证书或系统配置，先确认它作用于测试环境而非生产环境。

## 6. 部署后检查

部署完成后，至少检查以下项目：

- 容器是否成功启动
- 健康检查接口是否正常
- Nginx 配置是否通过校验
- 面板是否可访问
- 现有服务是否被影响
- 当前部署对应的 Git commit 是什么

## 7. 项目特定约束

- 项目为 `easy_vpn`
- 服务端部署方式为 `Docker Compose`
- Nginx 在宿主机上做反向代理
- TCP 端口范围是 `2200-2299`
- 不要硬编码具体业务端口，端口由面板统一分配
- GitHub 为公开仓库，禁止提交以下敏感信息：
  - IP
  - 密钥
  - `.env`
  - 密码
  - 真实配置

## 8. 沟通规则

- 如果一次操作可能碰到生产环境，必须先停下来确认。
- 如果当前状态存在非显而易见风险，要先说明风险再继续。
- 如果用户已经给出明确规则，应优先写入 `.codex/memory.md` 或本文件，避免再次遗忘。
